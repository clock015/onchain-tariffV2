# 原子贸易核心治理系统 (Atomic Trade Core System)

## 1. 系统概述

本系统是一个基于以太坊的去中心化贸易与治理集成方案。其核心逻辑是将商业贸易行为与链上治理权进行深度耦合：治理权重不通过二级市场购买获得，而是通过真实的贸易贡献（交易额）产生。系统通过“双重共识治理”与“年度归一化权重”机制，旨在平衡买卖双方（消费端与供应端）的长期利益博弈。

## 2. 核心架构

系统由三个主要层次组成：

*   **业务层 (Business Layer)**：由 `Market.sol` 和 `TradeExecutor.sol` 组成，负责处理资金流转、商家准入及税收逻辑。
*   **权益层 (Rights Layer)**：由 `ProportionalElection.sol` 及其配套工厂组成，负责治理权代币（SeatToken）的铸造、年度权重归一化及滑动窗口管理。
*   **决策层 (Decision Layer)**：基于 `FinalGovernor.sol`，实现双重共识投票逻辑、权限管控与指令执行。

## 3. 核心运行原理

### 3.1 贸易分配模型 (100%-90%-9%-1%)
当一笔交易在 `Market` 合约中发生时，系统对总交易额（Amount）进行如下分配：
1.  **商家结算 (90%)**：由 `TradeExecutor` 转发至商家指定的业务合约或地址。
2.  **税金池 (9%)**：暂存于 `Market` 合约内。该部分资金对应产生的“逆差积分”与“顺差积分”。
3.  **国库费 (1%)**：直接拨付至协议金库（Vault）。
4.  **权益铸造**：以国库费（1%）为基准，系统为买卖双方各铸造等值的权益代币 `SeatToken`。

### 3.2 积分对冲与退税机制
系统鼓励账户在生态内实现贸易平衡（即同时具备买家与卖家身份）。
*   **买方逆差积分 (Deficit Points)**：通过消费行为积攒，代表资金流出额度。
*   **卖方顺差积分 (Surplus Points)**：通过销售行为积攒，代表资金流入额度。
*   **运行逻辑**：用户积攒的“逆差积分”与“顺差积分”可以进行对冲。用户调用 `claimTaxRefund` 时，系统计算 `min(逆差积分, 顺差积分)`，销毁对冲部分的积分，并从税金池中退还等额的代币（USDC）。

### 3.3 治理权重归一化与滑动窗口
为了防止治理权过度膨胀或被早期高频交易者永久垄断，系统实施了以下机制：
*   **年度归一化**：每一年度（365天）产生的总投票权重被锁定为 **100 票**。无论该年度总交易额规模如何，个人权重始终等于 `(个人代币持有量 / 该年度总发行量) * 100`。
*   **5年滑动窗口**：仅最近 **5 年** 内产生的权益代币具备投票权。全系统的总治理权重上限恒定为 **500 票**（$100 \text{ 票/年} \times 5 \text{ 年}$）。
*   **权重更替**：随着时间推移，旧年度的代币会自动移出活跃窗口，确保治理权始终由近期活跃的贡献者掌握。

### 3.4 双重共识投票模型与治理权限
在 `FinalGovernor` 中，一项提案的生效必须达成买卖双方的博弈平衡，并以此行使对整个协议的高度控制权。

#### **双重共识逻辑**
*   **计算公式**：`有效赞成票 = min(买方赞成总票数, 卖方赞成总票数)`。
*   **博弈平衡**：该模型确保了任何修改必须同时获得买方群体（逆差端）和卖方群体（顺差端）的多数支持。单方面的利益主张（如仅靠卖家推动涨价）无法强行通过提案。

#### **治理权限范围**
`FinalGovernor` 作为系统的最高权力机构，拥有包括但不限于以下权限：
*   **市场准入与驱逐权**：有权通过 `kickMerchant` 指令强行踢出违规商家，并没收其保证金。
*   **协议参数调节权**：调整交易挑战窗口期（Challenge Period）、设置手续费金库地址、修改治理门槛等。
*   **逻辑升级权**：授权 UUPS 代理合约的升级，决定协议未来功能的迭代方向。
*   **资产处置权**：管理国库资金的分配与使用。

## 4. 安全与技术特性

*   **灵魂绑定属性 (Soulbound)**：`SeatToken` 限制了 `transfer` 函数。治理权仅能通过贸易贡献获得，不可转让，从而防止了治理权与实际业务脱钩。
*   **UUPS 可升级性**：核心合约均采用 UUPS 代理模式，支持逻辑的平滑升级。
*   **重入保护**：使用 `ReentrancyGuardTransient`（基于 EVM 坎昆版本 TSTORE 指令）在保证安全的同时优化 Gas 消耗。
*   **商家挑战机制**：引入基于保证金的挑战窗口期。任何人可发起挑战，最终由治理模块（Governor）根据事实进行裁决。

## 5. 开发与测试环境

本工程使用 **Foundry** 框架开发：

*   **编译器版本**: `solc 0.8.20+`
*   **EVM 版本**: `cancun` (必须支持，以满足瞬态存储指令)
*   **依赖库**: OpenZeppelin Contracts & Upgradeable v5.x

### 运行测试
```bash
# 执行完整测试逻辑（包含业务流、时间滑动窗口、双重共识验证）
forge test --evm-version cancun -vv
```
